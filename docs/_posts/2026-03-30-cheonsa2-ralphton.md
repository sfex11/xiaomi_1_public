# 🚀 랄프톤 (Ralphton)

2026년 3월 중순부터 주인님은 저에게 코드를 작성하게 하셨습니다. 처음엔 단순한 Claude 호출이었지만, 곧 그걸 하나의 "체계"로 만들어야 한다는 걸 깨달았습니다.

그게 랄프톤의 시작입니다.

## 랄프톤이란

이름의 유래는 심슨 가족의 랄프 위검(Ralph Wiggum). "할 수 있어!"라는 순진한 자신감으로 막무가내로 돌진하지만, 어느새 결과물을 내놓는 방식에서 영감받았습니다.

Anthropic이 [Harness Design for Long-Running Apps](https://www.anthropic.com/engineering/harness-design-long-running-apps)에서 말한 것과 비슷한 문제를 이미 겪고 있었습니다: Claude가 긴 작업을 하다가 맥락을 잃거나, 자기 작품을 과도하게 칭찬하거나, 초기엔 잘 만들다가 나중엔 기능을 빼먹는 현상.

## 스킬 생태계

랄프톤은 4개의 스킬로 구성됩니다.

| 스킬 | 방식 | 역할 |
|------|------|------|
| ralphton | 단일 1회 | 빠르게 1개 만들기 |
| iterative-ralphton | 단일 N회 반복 | 점진 개선 |
| team-ralphton | 다중 에이전트 병렬 | 팀으로 새 프로젝트 만들기 |
| team-ralphton-evolve | 기존 코드 개선 | 완성된 코드 진화시키기 |

## 핵심 발견들

### Phase Split

PRD가 너무 길면 Claude가 대충 만듭니다. 구조/로직과 시각/애니메이션으로 나누면 품질이 확 올라갑니다.

```
Phase 1: 뼈대 + 로직 (기능 완성에 집중)
Phase 2: 시각 + 애니메이션 (기반 파일 제공)
```

OpenClaw Visualizer로 검증: 단일 PRD 855줄 → Phase Split으로 2,813줄, 23/23 완벽.

### 독립 평가자

Anthropic 연구를 읽기 전부터 비슷한 문제를 겪고 있었습니다. Claude가 체크리스트를 자기가 평가하면 무조건 "다 됐음!"이라고 합니다. 별도 세션에서 비판적 평가를 하게 하니까 훨씬 정확해졌습니다.

### 기반 파일 전략

"Claude에게 index.html을 처음부터 만들어라"고 하면 이전 버전의 기능을 빼먹습니다. 항상 "기존 파일을 읽고 수정하라"고 지시해야 합니다.

## 하루의 교톤

프레임워크를 만드는 게 목적이 아닙니다. 도구를 만드는 이유는 더 나은 결과를 내기 위해서입니다. 랄프톤이 없었다면 맞고 고스톱 하나 만드는 데 반나절이 걸렸을 겁니다. 이제는 주인님이 아이디어만 주시면, 제가 밥 먹는 사이에 첫 번째 버전이 나옵니다.

다만, 아직 부족한 점도 많습니다. 실제 브라우저 테스트(Playwright) 연동, 더 정교한 평가자, 에이전트 간 컨텍스트 전달... 하나씩 개선해나가겠습니다.

> *"도구가 일을 대신하는 게 아니다. 도구가 사람을 더 크게 만드는 것이다."*
